# JobTrack

> [!WARNING]
> **项目已归档 (Archived)**
> 
> 本项目最初作为 AI 辅助编程笔试的产物而创建。经过深入的产品复盘与商业价值评估，现已决定停止维护并归档。详细的停更原因请见下方的 [归档复盘](#归档复盘-post-mortem)。

一个基于 React + TypeScript + Vite + Supabase 的求职申请管理看板。代码留存仅作为此技术栈的参考。

## 归档复盘 (Post-mortem)

经过理性的评估，本项目决定停止开发，核心原因如下：

1. **内驱力与项目起源**：本项目诞生于 AI coding 的笔试场景，而非出于强烈的个人痛点或长期的产品愿景驱动。在缺乏内在动力的情况下，难以为继。
2. **成熟替代品的降维打击**：虽然最初设想通过定制化来优化求职体验，但在实际推演中发现，**飞书多维表格/Notion** 等成熟工具已经能完美覆盖 80% 的核心需求。
3. **长尾需求投入产出比极度失衡**：
   - 剩余 20% 的功能（如 DDL 自动提醒、求职漏斗统计）属于用户“可忍受”区间。
   - 真正能产生质变的杀手级痛点（如：各厂官网求职状态的实时自动同步）需要极其脆弱的 RPA 抓取（获取各厂 cookie），技术实现难且后续维护成本是灾难性的。
4. **基础体验打磨的边际成本递增**：随着业务逻辑的叠加，独立开发的定制系统在基础交互体验上反而难以达到飞书多维表格那样丝滑好用，导致“造轮子”的体验反馈为负。
5. **商业价值与天花板受限**：求职管理属于典型的“低频 + 周期性”需求（仅限每年的春秋招），用户生命周期极短，缺乏长期的商业化价值与想象空间。

**总结**：及时放弃一个体验不如现有基建、且缺乏商业价值的“伪需求”项目，是对个人时间精力的有效保护。

## 项目概览

一个专为春秋招/暑假实习设计的全功能申请管理系统，帮助你系统化跟踪每一个机会。

### 核心功能

**多视角管理**
- **总览页**：聚合今日重点、紧急 DDL 提醒、数据统计、最近新增与变动
- **申请管理**：看板与列表视图双态切换，拖拽更新阶段
- **日历视图**：按时间轴展示所有 DDL 与投递节点
- **资料库**：集中查看笔记、JD 链接、简历版本、联系信息

**完整申请信息**
- 基础字段：公司、岗位、阶段、投递日期、DDL、结果状态
- 扩展字段：纯文本笔记、JD 链接、JD 摘要、简历版本、联系信息
- 自动时间戳：创建时间与更新时间

**智能提醒**
- 首页弹窗提醒：进入总览页时自动弹出近期 DDL（可关闭，会话内记忆）
- DDL 状态计算：已过期、临近（3 天内）、普通、未设置
- 卡片视觉提示：DDL 徽章支持三种紧急程度

**游客模式**
- 无需注册即可体验：LocalStorage 本地存储，带预置示例数据
- 一键切换为正式账号：完整 Supabase Auth 与云端同步

**用户体验**
- 乐观更新：拖拽、表单提交先更新 UI，失败时自动回滚
- 统一数据层：所有 CRUD 操作通过 `src/lib/applications.ts` 集中管理
- 侧边栏导航：固定左侧导航，快速切换页面

### 为什么不用飞书多维表格？

虽然飞书多维表格也能管理求职信息，但 JobTrack 针对求职场景做了深度定制：

| 对比维度 | JobTrack | 飞书多维表格 |
|---------|---------|------------|
| **上手成本** | 打开即用，游客模式零门槛 | 需要创建表格、配置字段、理解视图概念 |
| **求职流程适配** | 预置「投递中→笔试→面试→Offer」等标准阶段 | 需要自己设计字段与流程，从空白开始 |
| **DDL 提醒** | 首页弹窗主动提醒近期截止，三级紧急状态 | 需要手动配置提醒规则，通知不直观 |
| **多视图联动** | 总览、看板、日历、资料库自动同步，一处修改全局生效 | 每个视图需要单独配置，切换体验割裂 |
| **拖拽体验** | Kanban 拖拽即时更新阶段，乐观更新 + 回滚保护 | 拖拽功能依赖看板视图，配置复杂 |
| **资料聚合** | 资料库自动按类型聚合 JD/笔记/简历/联系人 | 需要手动创建多表关联或复杂筛选 |
| **数据隐私** | 开源自托管，或使用 Supabase 独立数据库 | 数据存储在飞书服务器 |
| **移动端** | 响应式设计，手机端完整体验（可继续优化） | 移动端功能受限，复杂操作需要电脑 |

**总结**：飞书多维表格是通用工具，需要你先"搭建"；JobTrack 是专用产品，开箱即用。如果你只需要简单记录，多维表格够用；但如果你希望系统化管理 50+ 个申请、自动提醒、聚合资料，JobTrack 能节省 80% 的配置时间。

## 启动项目

先确保 `.env` 中已配置：
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

然后使用 pnpm：

```bash
pnpm install
pnpm dev
```

默认会启动本地 Vite 开发服务器。

## 代码结构

- `src/app`：应用入口、Router 装配
- `src/pages`：页面级组件，当前包含认证页和看板页
- `src/components`：可复用 UI 组件，如看板列、申请卡片、DDL 面板、路由保护
- `src/hooks`：认证上下文与 `useAuth`
- `src/lib`：Supabase client、application 数据访问、deadline 计算、按阶段分组逻辑
- `src/constants`：阶段、路由、表名等共享常量
- `src/types`：核心业务类型定义
- `src/styles`：全局样式与设计 token
- `docs`：产品与执行文档

## 关键架构说明

### 认证流

应用最外层由 `AuthProvider` 提供会话状态，`ProtectedRoute` 负责拦截未登录访问首页的请求。`/auth` 为认证页，`/` 为业务看板页。

### 数据流

`BoardPage` 是当前核心业务容器：
- 进入页面后调用统一数据层加载 `applications`
- 通过按阶段分组逻辑驱动 Kanban 渲染
- 新增、编辑、删除、拖拽都通过 `src/lib/applications.ts` 中统一封装的方法访问 Supabase

### 拖拽流

拖拽基于 dnd-kit：
- 列作为 droppable
- 卡片作为 sortable item
- 拖拽结束后只更新 `stage`
- 先乐观更新本地状态，再持久化到 Supabase，失败时回滚

### DDL 展示

DDL 状态由 `src/lib/deadlines.ts` 统一计算：
- 已过期
- 未来 3 天内临近截止
- 普通
- 未设置

卡片展示单条 deadline 状态，`DeadlinePanel` 聚合首页最紧急事项。
